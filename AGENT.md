# Compression Agent Instructions

## Goal

Improve the lossless compression algorithm implemented in `src/core.rs`. The aim is to reduce the average compression ratio (lower is better — a ratio of 0.5 means the output is half the size of the input) across a diverse set of sample files, while keeping compress and decompress times reasonable.

Each attempt is a separate pull request to `main`. The PR description must be **only** the raw output of `benchmark.sh` — five floating-point numbers on separate lines. This is used to track progress over time.

---

## What You Can Change

**Only `src/core.rs`.** All other files are off-limits unless you have a specific reason and ask first.

You may not add new Rust crate dependencies without permission.

---

## Project Structure

```
src/core.rs          ← the only file you should edit
src/main.rs          ← CLI wiring; do not touch
benchmark.sh         ← runs tests, builds, compresses/decompresses every sample, prints results
fetch_samples.sh     ← downloads public-domain sample files (run once to populate samples/)
samples/             ← input files for the benchmark (git-ignored, generated locally)
outputs/             ← compressed files written by benchmark.sh (git-ignored)
debug.csv            ← per-file breakdown written by the last benchmark run (local only, not committed)
```

---

## The Interface

The two public functions in `core.rs` must keep these exact signatures:

```rust
pub fn compress<S: Read + Seek, W: Write + Seek>(mut input: S, mut output: W) -> io::Result<()>
pub fn decompress<S: Read + Seek, W: Write + Seek>(mut input: S, mut output: W) -> io::Result<()>
```

Compress reads from `input`, writes compressed bytes to `output`. Decompress does the reverse. The decompressed output must be a bit-for-bit match of the original — the benchmark verifies this with `cmp` and will fail if it isn't.

## Sample Files

Run `bash fetch_samples.sh` once to populate `samples/`. It downloads public-domain files and derives less-compressed versions of each using ffmpeg. You should have a mix of:

| Type | Examples | Expected behaviour |
|---|---|---|
| Plain text | Project Gutenberg books | Compresses well — ratio well below 1.0 |
| BMP images | JPEG sources converted to raw RGB | Compresses well — large uniform regions |
| WAV audio | OGG/FLAC sources converted to PCM | Compresses moderately |
| Structured data | CSV/JSON (Gutenberg catalog, NASA catalog) | Compresses well — repetitive structured text |
| Binary executable | The compiled benchmark binary | Partially compressible structured binary |
| PNG images | JPEG sources converted losslessly | Moderate compression |
| JPEG images | NASA photographs | Already compressed — ratio ≈ 1.0 |
| OGG/FLAC audio | Classical music recordings | Already compressed — ratio ≈ 1.0 |
| MP4 video | Short films | Already compressed — ratio ≈ 1.0 |
| Random binary | Generated by benchmark.sh | Incompressible — ratio > 1.0 |

## Running the Benchmark

```bash
bash benchmark.sh
```

This will:
1. Run `cargo test` — both tests must pass
2. Build a release binary
3. Generate any missing random sample files
4. Compress and decompress every file in `samples/`, verifying:
   - Bit-perfect roundtrip
   - Compressed output does not exceed 1.5× the original size
   - Compressing the same file twice produces identical output (determinism)
5. Print five numbers to stdout (all other output goes to stderr):

```
<overall size-weighted ratio>        # total_compressed / total_original — lower is better
<compressible files ratio>           # same metric, text/BMP/WAV/binary only
<already-compressed files ratio>     # same metric, JPEG/OGG/MP4/PNG only — should stay near 1.0
<average compress time ms>           # wall-clock milliseconds per file
<average decompress time ms>         # wall-clock milliseconds per file
```

The split between compressible and already-compressed files is the clearest signal of algorithm quality: the second line measures genuine compression, the third measures whether the algorithm gracefully handles data it can't compress.

Any failure — timeout, mismatch, excess expansion, or non-determinism — causes the benchmark to exit immediately with a non-zero code. A timeout is treated the same as a correctness failure.

`debug.csv` is written after each run with per-file breakdowns. Read it before starting — it shows exactly where the current algorithm leaves gains on the table.

## Resource Constraints

The benchmark runs on a modern consumer machine with a reasonable amount of RAM. There is no hard limit, but the implementation should be practical — loading an entire multi-hundred-megabyte file into memory at once is fine, but designs that require holding multiple large copies simultaneously (e.g. input + output + several auxiliary buffers all at full file size) will run into trouble on the larger samples and should be avoided. As a rough guide, aim for peak memory usage no more than 3–4× the size of the file being processed.

CPU usage is similarly unconstrained but should be proportional to input size — an O(n²) algorithm that runs acceptably on small files may time out on the 100MB+ samples. If a design is fundamentally slow, consider whether it can be bounded (e.g. capping a search window) rather than abandoned.

---

## Approach

The goal is to experiment and discover novel or unconventional approaches to compression — not to produce a clean Rust reimplementation of an established algorithm. Known techniques (LZ77, Huffman, BWT, arithmetic coding, etc.) are useful background to understand and draw from, but the interesting work is in combining ideas in new ways, finding approaches that exploit specific properties of the benchmark corpus, or exploring directions that the literature hasn't fully settled.

The current `core.rs` on `main` is the best implementation found so far. Study `debug.csv` from the last benchmark run to understand where gains are being left on the table — which file types compress well, which don't, and where the time is going.

You should build from first principles rather than wrapping an existing library, and avoid copying implementations verbatim. Negative results (approaches that don't improve the benchmark) are still worth attempting — the PR history is the record of what's been tried.
